Kubernetes Homelab

After running everything on Proxmox for a while, I wanted to learn Kubernetes properly. Not managed Kubernetes where the hard parts are hidden, but the real thing on bare metal. So I built a cluster from scratch using Talos OS.

Why Talos

Talos OS is a minimal, immutable Linux distribution built specifically for Kubernetes. No SSH, no shell, no package manager. You manage everything through an API. It sounds limiting until you realize how much less can go wrong. The OS is declarative and versioned, just like the workloads running on it.

Networking

kube-vip handles the control plane VIP and load balancer services. No need for an external load balancer or MetalLB. It runs as a static pod on the control plane nodes and uses ARP or BGP depending on your setup.

Traefik as the ingress controller. It handles routing, TLS termination, and integrates well with cert-manager for automatic certificates.

Storage

Longhorn for distributed block storage. It replicates data across nodes, handles snapshots and backups, and has a decent UI for when you need to see what's going on. Not the fastest option, but reliable and easy to manage.

Certificates

cert-manager handles TLS certificates automatically. Point it at Let's Encrypt (or your own CA), define some issuers, and it takes care of the rest. No more manual certificate renewal.

GitOps

Everything is deployed through FluxCD. The cluster state lives in a Git repo. Push a change, Flux picks it up and applies it. No more kubectl apply from my laptop. If I mess something up, I can see exactly what changed and roll it back.

Services

Same stuff I was running on Proxmox, now in containers on Kubernetes:

  • Nextcloud for files, calendar, contacts
  • Jellyfin for media
  • Vaultwarden for passwords
  • Pocket ID for SSO
  • Home Assistant for home automation
  • Immich for photo backups
  • Servarr stack for media management

What I learned

Running Kubernetes at home is a lot of work upfront. But once it's set up, the GitOps workflow is hard to beat. I can rebuild the entire cluster from the repo if I need to. Upgrades are less scary when you can just spin up a new node and drain the old one.
